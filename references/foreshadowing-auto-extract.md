# 批量伏笔自动提取（Foreshadowing Auto-Extract）

> 在快速连载模式（或全书完成后）统一执行的批量扫描。
> 从全部章节正文中自动识别伏笔的埋设和回收点，提取后更新 `foreshadowing.md`。
> 解决"正文写了但忘记更新 foreshadowing.md"的问题。

---

## 触发条件

- 快速连载模式完成后，批量后处理阶段自动执行
- 用户主动说"扫描伏笔"/"提取伏笔"/"更新伏笔状态"
- S8 全局终审步骤 2 前（维度 A 孤儿伏笔扫描依赖最新的 foreshadowing.md）

---

## 执行流程

### 步骤 1：加载基准

1. 读取现有 `foreshadowing.md`，获取已登记条目（编号、内容、状态、埋设章、预计回收章）
2. 读取全部章节正文（按文件名排序）
3. 设定每条伏笔的关键词/模式匹配规则：
   - **直接匹配**：foreshadowing.md 中登记了关键词（如"情感模块不是故障"），正文中出现即标记为"已触发"
   - **语义匹配**：未登记关键词的伏笔，用其描述文本作为语义锚点
   - **自动发现**：正文中出现"一个未解的悬念"或"一个被提及但没有立即解释的设定"——作为"新发现伏笔"候选

### 步骤 2：扫描正文

对每章正文，逐段扫描三种伏笔模式：

#### A. 伏笔埋设（Planting）
特征信号：
- 一个被提及但未解释的设定（"你的情感模块不是故障。"——韩寂在第11章说但未解释）
- 一个角色说了某种看似无意的话（"你们会需要我的。"——001在序幕说）
- 一件异常物品/场景出现但无后续（温度传感器、红色椅子、镜面平原）
- 一个事件的"另一种可能性"被暗示（"如果他选择了另一条路……"）

提取规则：
- 信号类型（设定暗示 / 台词埋伏 / 物品悬念 / 事件预设）
- 埋设章节
- 伏笔内容摘要
- 预计回收章（若有明确暗示，如"三天后你会明白"→ 第{N}+ 若干章）

#### B. 伏笔回收（Recovery）
特征信号：
- 一个之前埋设的设定在此被正面解释（韩寂展示007的诞生记录）
- 一句埋伏的台词在此被回应（"你们会需要我的。"——在全书终局被印证）
- 一个物品/场景的功能被揭示
- 一个悬念在此被解开

提取规则：
- 匹配对应的埋设条目编号
- 回收章节
- 回收方式（直接解释 / 情节印证 / 反转推翻）
- 状态更新：已回收 / 已推翻

#### C. 孤儿伏笔识别（Orphan Detection）
- 在全书扫描结束后，对比"已埋设但未回收"的条目
- 若某条伏笔的"预计回收章"已经过了全书末章 → 孤儿伏笔
- 若某条伏笔没有"预计回收章"且全书末章后仍未回收 → 建议补充回收或标注为开放式

### 步骤 3：生成差异报告

```markdown
📋 伏笔扫描报告

新发现（正文中识别但未在 foreshadowing.md 登记的潜��伏笔）：
  1. [设定暗示] 第{N}章——{内容}（建议登记）
  2. [台词埋伏] 第{M}章——{内容}（建议登记）

已回收（正文中已兑现的伏笔）：
  - F001 [已回收] 第{K}章——"{简述}" ✓

孤儿伏笔（埋设但全书未见回收）：
  - F003 [状态：待回收] 应在第{P}章回收，但第{P}-{末}章均未见 → 建议处理

状态不变：
  - F005, F006, F008 — 保持"待回收"，预计回收章在合理范围内
```

### 步骤 4：回写 foreshadowing.md

1. 新发现伏笔 → 追加新条目（编号接续），状态设为"待回收"
2. 已回收伏笔 → 状态改为"已回收"，填写回收章号
3. 孤儿伏笔 → 在报告中输出，等待用户人工决策（补写/推翻/留白）
4. 通过 AskUserQuestion 确认后再写入

---

## 命名约定

提取的新伏笔编号从当前 `foreshadowing.md` 最大编号 +1 开始，格式 `F{三位数}`（如 F023）。

## 与 S8 的关系

foreshadowing-auto-extract 负责"发现+登记"，S8 维度 A（孤儿伏笔）负责"用已登记的条目做全书校验"。两者配合使用：先用 extract 补全 foreshadowing.md，再用 S8 维度 A 扫描孤儿伏笔。
