# 怒笔（FuryPen）小说锻造系统

## 概述

NovelForge 是基于 WorkBuddy Skill + Memory 架构的长篇小说一致性写作系统。核心解决三个问题：上下文窗口装不下全书、AI 无持久状态、无一致性闸门。

架构 = 结构化故事圣经(bible/) + 读-写-校验循环(S2→S3→S4→S5) + 类型定制一致性闸门 + 增强工具(S6/S7/S8)。v3 新增快速连载模式（批量写后改）+ 草稿精修两遍 + S8 全局终审扩展。

**各家之长**：蛙趣拼文(8模块+伏笔追踪+角色成长轨迹)、NovelCrafter(Codex百科+场景挂载按需注入)、NovelAI(Lorebook关键词触发)、Sudowrite(扩写/重写)、AI_NovelGenerator(多阶段生成+ChromaDB向量+自动审查)。独创：热点方向菜单+Q&A引导初始化+类型分层校验闸门。

## 工作流判断

根据用户意图判断执行哪个子流程：

| 用户意图关键词 | 执行流程 |
|--------------|---------|
| "写小说"/"创建小说"/"novel init"/"小说初始化"/"新建小说" | → S1 小说初始化 |
| "全部写完再改"/"快速连载"/"先写后改"/"批量写" | → 进入快速连载模式（run-state 标记 fast + draft），S2→S3（跳过审阅）→跳过S4→跳过S5 → 全部写完后批量后处理 |

## 执行编排与状态机（v2 新增，所有 Skill 遵守）

NovelForge 是长线工程，小说往往跨数十个会话。Agent 必须保证**任意时刻中断都能准确恢复**。本段规定跨 Skill 的编排纪律。

### 1. run-state.md 是中枢

- 项目创建时（S1 步骤8）在 `tracking/` 下生成 `run-state.md`（用 `assets/bible-templates/run-state.md` 模板）
- **每次进入/退出任意 Skill 前，先读 `run-state.md` 确认当前阶段；退出前更新它**
- 用户新开会话说"继续写/接着写"时，Agent 第一动作：读 `run-state.md` → 按"下一动作"字段续执行，不重新问已确定的设定

### 2. 主循环自动链式推进

S2→S3→S4→S5 是一章的标准闭环。**默认自动串联**，无需用户每步确认：

```
用户："写第N章"
  → S2 注入 → S3 撰写(含写前自检) → S4 提取回写 → S5 审查
       ├─ S5 通过 → 询问"继续写第N+1章？" / 检查大纲续展 / 更新 run-state
       └─ S5 HARD违规 → 触发 S7 重写 → 回到 S4→S5（更新 run-state 的"待处理"）
```

- 只有 **S3 用户审阅**、**S5 硬违规修复选择**、**大纲续展确认** 需要用户决策
- S2/S4 的内部步骤全自动，不弹问
- 每完成一章，更新 `run-state.md` 的"当前章节/阶段/下一动作/累计统计"

### 3. 非线性写作支持

- 用户可指定任意章节号写："写第12章"即使第3-11章未写
- 此时 S2 仍按第12章的章节卡注入（取 outline 中第12章卡 + 截至当时的 bible 状态）
- 风险：第12章可能用到第3-11章才该揭示的信息 → S5 信息一致性层会标黄，S8 全局终审会抓"跨章信息泄漏"
- Agent 应在非线性写作时于 `run-state.md` 标注"跳跃写作"，提示后续补写中间章时复用已写内容

### 5. 快速连载模式（v3 新增）

当用户说"全部写完再改"/"快速连载"/"先写后改"/"批量写"时：

1. 在 `run-state.md` 中标记 `写作模式: fast`、`写作阶段: draft`
2. S2→S3→S4→S5 循环中，S3 跳过用户审阅（fast_mode=true），S4 不立即执行回写（改为积累到 pending）
3. S5 审查也跳过（快速模式下一致性由作者在 polish 阶段统一审查）
4. 连续写完全部或指定批次章节后，进入批量后处理：
   - 跑批量 S4：一次性提取全部章节的事实，集中回写 bible（角色/伏笔/数值/时间线）
   - 跑批量 S5：对全部章节做一次性扫描审查
   - 跑 `references/foreshadowing-auto-extract.md`：自动从正文中提取伏笔埋设/回收，更新 foreshadowing.md
   - 询问用户：是否进入 **polish 阶段**（回头逐章精修改写）

### 6. 草稿/精修两遍模式（v3 新增）

写作阶段由 `meta.md` 的 `写作阶段` 字段控制：

- **draft（草稿阶段）**：
  - 速度优先：每章篇幅按 weight×0.7，描写从简，对话优先
  - 允许偶有表达粗糙，重点保证故事主线流畅、剧情不卡
  - 不跑 S4/S5——全部在批量后处理中一次完成
- **polish（精修阶段）**：
  - 质量优先：每章按 full length 重写或精修
  - 细修描写、对话节奏、伏笔密度
  - 逐章跑 S5 审查
- 切换指令："开始精修"/"进入 polish 阶段"/"回头改" → 将 meta.md 的`写作阶段`改为 polish，对已写章节逐一触发 S7 修订

### 7. 大纲续展自动检查

- S5 步骤6（询问是否继续）前，先检查 `outline.md` 剩余"待写作"章节卡 ≤ 2
- 若是 → 先跑 `references/outline-extension.md` 续展下一段落，再继续写

- S5 步骤6（询问是否继续）前，先检查 `outline.md` 剩余"待写作"章节卡 ≤ 2
- 若是 → 先跑 `references/outline-extension.md` 续展下一段落，再继续写

## S1: 小说初始化（一次性，创建整个项目骨架）

### 步骤 0: 热点方向选择

1. 读取 `references/hot-directions.md`，获取完整菜单数据
2. 使用 AskUserQuestion 工具呈现4大类热点方向：
   - A: 网文商业热点（规则怪谈/都市异能/末世囤货/女性悬疑）
   - B: 融合创新方向（赛博仙侠/科幻仙侠/中式克苏鲁/废土朋克）
   - C: 严肃文学向（气候小说/推想回忆录/暗黑学院/心理惊悚）
   - D: 现实观照向（职场逆袭/科技自强/时代与个人/传统文化）
3. 用户选大类后，再用 AskUserQuestion 追问具体子方向（4选项）
4. 如用户选"Other"自定义，记录用户输入的类型名称
5. 将选定类型名称记为 `{GENRE}`，后续步骤全部基于此类型

### 步骤 1: 加载类型配置

1. 读取 `references/genre-config/{GENRE}.md`
2. 如该类型文件不存在，读取 `references/genre-config/通用.md` 作为兜底
3. 从配置中提取三个关键数据：
   - `required_modules`: 必须生成的 bible 模块列表
   - `check_priority`: 一致性校验优先级（S5 闸门配置）
   - `init_qa_custom`: 各阶段的定制追加问题（叠加在通用问题之上）

### 步骤 2: Q&A Phase 1 — 核心设定

产出: `meta.md` + `core-rules.md`(如需要)

通用问题（每个用 AskUserQuestion 逐个追问）：
1. "请给小说取一个名字（或暂定名）"
2. "一句话概括核心冲突/故事内核是什么？"
3. "叙事视角：第一人称限知？第三人称全知？第三人称限知？多视角交替？"
4. "基调：黑暗压抑？热血燃？温情治愈？悬疑紧张？荒诞幽默？"
5. "预计篇幅：短篇(5万字内)？中篇(5-20万)？长篇(20-50万)？超长(50万+)？"

叠加 genre-config 的 `init_qa_custom.phase1_extra` 问题（如 规则怪谈 会追加轮回结构/进入机制/死亡条件/规则范式/规则冲突优先级/成长系统等问题）。

每个问题获得回答后，立即写入对应文件。使用 `assets/bible-templates/meta.md` 作为模板骨架，填充用户回答。

### 步骤 3: Q&A Phase 2 — 人物设计

产出: `characters/_index.md` + 每个角色独立文件 `characters/{角色名}.md`

通用问题：
1. "主角的名字和性格？用三个词锁定性格内核（如'隐忍/偏执/重情'）"
2. "主角靠什么活下来/取胜？金手指？智力流？信息差？"
3. "主角开局知道多少信息？完全无知？部分已知？"
4. "2-3个核心配角/NPC：名字+立场+秘密"

叠加 genre-config 的 `init_qa_custom.phase2_extra`。

每个角色用 `assets/bible-templates/character.md` 模板生成独立文件，包含：性格三词、外貌口癖、动机与弧光、关系网、已知信息集、存活状态、能力/技能。

### 步骤 4: Q&A Phase 3 — 首章/首个副本设计

产出: `outline.md`(章节卡) + `instances/_index.md` + 第一个副本文件(如 规则怪谈)

通用问题：
1. "第一个场景/副本的核心冲突是什么？"
2. "本章出场人物有哪些？"
3. "本章结尾要留下什么悬念？"

叠加 genre-config 的 `init_qa_custom.phase3_extra`（如 规则怪谈 追问副本规则集/通关条件/伏笔埋设）。

用 `assets/bible-templates/outline.md` 模板生成章节卡，每章包含：定位、目标、悬念级别、伏笔操作、出场角色列表。
如需要副本，用 `assets/bible-templates/instance.md` 模板。

### 步骤 5: Q&A Phase 4 — 长线悬念

产出: `worldbuilding/` 系列文件 + `timeline.md`

通用问题：
1. "故事的最终真相/世界观底层秘密是什么？（读者慢揭开，但作者现在确定）"
2. "预计总共多少章/多少个副本？"
3. "故事结局走向：胜利？悲剧？开放式？"

叠加 genre-config 的 `init_qa_custom.phase4_extra`。

用 `assets/bible-templates/` 中对应的模板生成文件。

### 步骤 6: 生成 Lorebook

1. 从已建设定中提取关键名词（人物名、地点名、规则名、道具名等）
2. 为每个关键名词建立映射规则："当关键词 X 出现时，注入对应设定文件"
3. 用 `assets/bible-templates/lorebook.md` 模板生成 `bible/lorebook.md`

### 步骤 7: 生成文风指南

追问文风偏好：
1. "语言风格：华丽修辞？克制白描？口语化？诗意散文？"
2. "节奏偏好：快节奏事件堆叠？慢节奏心理刻画？交替节奏？"
3. "有什么绝对禁用的表达？（如'不禁''竟然'等AI味词汇）"

用 `assets/bible-templates/style-guide.md` 模板生成。

### 步骤 8: 创建项目目录

在当前工作目录下创建完整的项目结构：

```
{小说名}/
├── bible/          (所有生成的 bible 文件)
│   ├── meta.md
│   ├── genre-config.md        (从 genre-config 复制，作为项目专属配置)
│   ├── core-rules.md          (如类型需要)
│   ├── outline.md
│   ├── characters/
│   │   ├── _index.md
│   │   └── {每人.md}
│   ├── worldbuilding/
│   │   ├── locations.md
│   │   ├── rules.md
│   │   ├── timeline.md
│   │   └── lore.md
│   ├── instances/             (如类型需要)
│   │   ├── _index.md
│   │   └── {每副本.md}
│   ├── foreshadowing.md
│   ├── progression.md         (如类型需要)
│   ├── lorebook.md
│   └── style-guide.md
├── chapters/      (空目录，待写入)
├── tracking/
│   ├── progress.md
│   ├── consistency-log.md
│   ├── rewrite-history.md
│   ├── run-state.md            (v2: 状态机，断点续写中枢)
│   └── global-audit-log.md     (v2: S8 全局终审日志)
```

只创建 `required_modules` 中列出的模块文件，不在 required_modules 中的跳过。

**v2 强制文件（不受 required_modules 限制，始终创建）**：
- `tracking/run-state.md`：用 `assets/bible-templates/run-state.md` 模板生成。断点续写中枢，S1 完成后立即写入书名/类型/项目路径/当前阶段=初始化完成/当前章节=第1章/下一动作=写第1章
- `tracking/global-audit-log.md`：用 `assets/tracking-templates/global-audit-log.md` 模板生成，为 S8 全局终审预留

### 步骤 9: 确认

1. 展示 bible 摘要（每个文件一句话概要）
2. 使用 AskUserQuestion 让用户确认："项目骨架已生成，是否需要调整？"
3. 如用户要求调整，修改对应文件
4. 确认后，写入 `.workbuddy/memory/MEMORY.md` 项目长期记忆，记录小说名/类型/核心设定

## S2: 上下文注入（每章循环第一步）

**触发条件**：用户说"写第X章"/"开始写章"/"续写"/"write chapter"，或 S3 自动接续。

### 步骤 1：确定章节号和项目路径

1. 从用户指令中提取章节号 `{N}`（如用户说"继续写"且无章节号，读 `run-state.md` 取"当前章节+1"）
2. 确定小说项目目录路径 `{PROJECT}`（优先从 `{PROJECT}/tracking/run-state.md` 读取；其次从 `.workbuddy/memory/MEMORY.md` 读取）
3. 读取 `run-state.md`，确认当前阶段不是"需重写"遗留下来的冲突状态；将"当前阶段"更新为 `S2注入`、"当前章节"更新为第{N}章
4. 如未找到项目路径，提示用户："请先运行 S1 初始化一个小说项目"

### 步骤 2：读大纲取章节卡

1. 读取 `{PROJECT}/bible/outline.md`
2. 找到"### 第{N}章"段落，提取：
   - 定位（开场/引入/转折/高潮/收束）
   - 目标
   - 悬念级别
   - 伏笔操作（埋设/推进/回收/无）
   - 出场角色列表
   - 一句话梗概
3. 如该章节卡不存在，提示用户："第{N}章尚未在大纲中定义，请先补充章节卡到 outline.md"

### 步骤 3：读出场人物档案

1. 读取 `{PROJECT}/bible/characters/_index.md`
2. 根据步骤 2 获取的出场角色列表，逐个读取 `{PROJECT}/bible/characters/{角色名}.md`
3. 从每个角色文件提取摘要信息（用于上下文组装）：
   - 性格三词
   - 当前存活状态
   - ★已知信息集（开局已知 + 后续获取）
   - 与其他出场角色的当前关系
   - 口癖/标志动作
   - 能力限制

### 步骤 4：读副本规则（如 规则怪谈/无限流类型）

1. 读取 `{PROJECT}/bible/genre-config.md`，确认类型
2. 如类型需要副本系统（规则怪谈），读取 `{PROJECT}/bible/instances/_index.md`
3. 找到当前章节所属副本：`出场章节` 范围包含第 {N} 章的副本
4. 读取 `{PROJECT}/bible/instances/{当前副本}.md`，提取：
   - 副本特有规则集（逐条）
   - 通关条件
   - 已建立事实
5. 读取 `{PROJECT}/bible/core-rules.md`，提取元规则和冲突优先级
6. 如非副本类型，此步骤跳过

### 步骤 5：Lorebook 全量关键词匹配注入

> v2 升级：从"只扫章节卡"升级为"扫本章已加载的全部上下文信号"，大幅降低漏触发。

1. 读取 `{PROJECT}/bible/lorebook.md`
2. **全量扫描**以下来源中的关键词（不止章节卡）：

4. **token 预算裁剪**（当组装内容预估超预算时按优先级保留）：

   - 目标：上下文提示词控制在可稳定生成 `{字数}` 字章节的范围内（一般 ≤ 全文生成的 1.5 倍预算）
5. 将组装完成的提示词传递给 S3，并更新 `run-state.md`（S2完成=是）

## S3: 撰写章节

**触发条件**：S2 上下文注入完成后自动接续，或用户说"写第X章"时（先执行 S2 再接 S3）。

### 步骤 1：读取章节参数

1. 读取 `{PROJECT}/bible/meta.md`，获取：
   - **写作阶段**：`写作阶段: draft`（快速写故事线，描写简略，节奏优先）或 `polish`（质量优先，每章精写细修）
   - **基准字数**：默认为 `meta.md` 中的每章目标字数（如 5000 字）
2. 读取 `outline.md` 第{N}章卡片，获取三个调节参数：
   - **`weight`（1-5）**：用于调节本章篇幅。1★ = 基准×0.5，3★ = 基准×1.0，5★ = 基准×2.0。若不填，按**悬念级别**自动推导：★=1★，★★=2★，★★★=3★，★★★★=4★，★★★★★=5★
   - **`chapter_role`（opening/normal/climax/finale）**：用于调节写作模式和篇幅。`opening` 需要额外描述性段落建立世界观；`climax` 需要更充分的动作/对话展开；`finale` 需要更充分的收束空间。`normal` 按标准模式处理
   - **`fast_mode`（true/false）**：若为 true，跳过步骤 4（用户审阅），直接写正式版。默认 false。当用户说"全部写完再改"或"快速连载"时可全局开启
3. 综合计算本章目标篇幅：基准字数 × weight 系数 × chapte_role 系数（opening=1.3×，climax=1.2×，finale=1.3×，normal=1.0×）× 写作阶段系数（draft=0.7×，polish=1.0×）

### 步骤 2：生成章节正文

1. 使用 S2 组装完成的上下文提示词
2. 直接以该提示词为指引，撰写第 {N} 章正文
3. 写作时严格遵守提示词中的所有约束（规则/已知信息/数值/文风/禁用词）
4. 确保章节节奏符合 style-guide.md 的模板（开头→中段→结尾）
5. 用 `assets/bible-templates/chapter.md` 的章节结构（场景标记 + 章末元数据注释）组织正文，便于 S4 提取与 S6 定位

### 步骤 2.5：写前自检（v2 新增，关键质量闸）

> 在写入草稿、交给用户之前，先按 `references/self-check-rules.md` 做一次内部预审查，就地修复 HARD 级问题，避免把明显违规稿交到用户眼前、也减少后续 S7 重写轮次。

1. 读取 `references/self-check-rules.md`
2. 取 S2 上下文提示词中的全部约束（已知信息集 / 规则条文 / 数值 / 伏笔 / 禁用词 / 文风），逐条对照刚生成的草稿
3. 输出内部**自检报告**（格式见 self-check-rules.md），标注每项 HARD/SOFT 结果
4. 若发现 HARD 违规或禁用词：
   - **就地重写问题段落**（保留无问题的精彩段落），重新自检
   - 最多自修复 2 轮；仍不通过则照常写入草稿交 S5 处理（不阻塞）
5. 自检通过后，将草稿中的禁用词清零、确认信息/规则无越界，再进入步骤 3

> 自检报告可一并在用户审阅时附带展示（标注"已自检通过"），提升用户信任。

### 步骤 3：写入草稿

1. 将生成的正文写入 `{PROJECT}/chapters/ch{N}-draft.md`
2. 文件开头标注草稿状态：

```
> 📝 草稿版本 — 待用户审阅确认
> 生成时间：{当前时间}
> 状态：draft
```

3. 正文内容直接写入

### 步骤 4：用户审阅（受 fast_mode 控制）

> 若本章 `fast_mode=true` 或用户指定了"快速连载"，则跳过本步骤，直接进入步骤 5。

1. 检查是否是快速模式：
   - 若本章 `outline.md` 第{N}章卡片标记 `fast_mode: true`，或 `run-state.md` 标记`快速连载中` → 跳过步骤 4 和步骤 3，直接写正式版（步骤 5）
   - 若为正常模式 → 继续步骤 2-3（写草稿），然后执行以下：
2. 向用户展示草稿全文（或关键段落摘要，如全文过长）
3. 使用 AskUserQuestion 询问："草稿已生成，请审阅。是否确认此版本？"
   - 选项 A："确认，正式定稿" → 进入步骤 5
   - 选项 B："需要微调" → 用户给出修改意见，微调后重新展示
   - 选项 C："不满意，重新生成" → 回到步骤 2 重新撰写（但保留 S2 上下文不变）
4. 微调时，只修改用户指定的段落/表达，不重写整章
5. 最多允许 3 次重新生成，超过后提示用户手动调整大纲或设定
6. 快速模式下，每完成 5 章或每卷结束时，可主动询问用户是否要切回正常模式审阅一次

### 步骤 5：写入正式版

1. 将确认后的正文写入 `{PROJECT}/chapters/ch{N}.md`
2. 文件开头标注正式状态：

```
> ✅ 正式版本 — 第{N}章定稿
> 定稿时间：{当前时间}
> 状态：final
```

3. 删除草稿文件 `ch{N}-draft.md`（如存在）
4. 通知用户："第{N}章定稿完成，接下来自动执行 S4 事实提取+回写"
5. 自动进入 S4 流程

## S4: 事实提取+回写

**触发条件**：S3 章节定稿后自动接续。**快速模式下延迟到批量后处理阶段统一执行**。

> **快速模式说明**：若 `run-state.md` 标记了`写作模式: fast`，S4 跳过立即执行。改为在"批量后处理"阶段一次性集中执行全景提取。具体做法：先在本章 `run-state.md` 的 `pending_extractions` 列表中记录"第{N}章待提取"，全部章节写完后统一调用"批量 S4"流程——一次性读取全部待提取章节正文，做 8 维度全景提取后集中回写 bible（角色/伏笔/数值/时间线/副本/Lorebook 一次性全量更新）。

### 步骤 1：读取提取规则

1. 读取 `references/extraction-rules.md`（本 Skill 的提取维度和回写映射）
2. 确认需要提取的维度列表：新角色/角色状态变化/已知信息更新/伏笔操作/数值变动/时间线事件/副本事实/Lorebook关键词

### 步骤 2：读取章节正式版

1. 读取 `{PROJECT}/chapters/ch{N}.md`（刚定稿的正式版）
2. 逐段扫描，识别提取信号

### 步骤 3：逐维度提取事实

按 extraction-rules.md 中定义的 8 个维度依次提取：

#### 3.1 新角色出现
- 扫描章节中新出现的人名
- 交叉比对 `characters/_index.md`，不在列表中的 = 新角色
- 提取：角色名/外貌/关系/性格特征/关键行为

#### 3.2 角色状态变化
- 扫描角色情绪/立场/关系/存活状态的明确变化
- 提取：角色名/变化类型/变化内容/导致事件/章节号

#### 3.3 已知信息更新 ★最高优先级
- 扫描角色获得了之前不知道的信息
- 提取：角色名/新信息内容/获取方式/章节号
- **此维度必须100%准确，直接影响闸门第2层**

#### 3.4 伏笔操作
- 埋设：扫描暗示性细节（异常描写/话中有话/规则暗示/异常道具）
- 回收：扫描之前伏笔的兑现/推翻/部分兑现
- 提取：伏笔内容/编号/章节号/类型/状态变更

#### 3.5 数值变动
- 扫描积分/技能等级/道具数量的明确变化
- 提取：变动项/变动前后值/原因/章节号

#### 3.6 时间线事件
- 扫描核心事件（战斗/发现/转折/死亡/进入/退出等）
- 提取：事件摘要/章节号/涉及角色/事件类型

#### 3.7 副本事实（规则怪谈专属）
- 扫描副本中确认的新规则事实/通关线索
- 提取：新事实/确认方式/影响范围/章节号

#### 3.8 新关键词 → Lorebook
- 扫描首次出现的名词（地点名/规则名/道具名/组织名）
- 提取：关键词/来源文件/注入方式

### 步骤 4：汇总提取结果

1. 将所有提取结果整理为结构化列表
2. 向用户展示提取摘要：

```
📋 第{N}章事实提取结果

新角色：{数量}个 — {角色名列表}
角色状态变化：{数量}条 — {摘要}
已知信息更新：{数量}条 — {角色名+信息摘要}
伏笔操作：埋设{X}条 / 回收{Y}条 / 推翻{Z}条
数值变动：{数量}条
时间线事件：{数量}条
副本事实：{数量}条（如适用）
新关键词：{数量}个
```

3. 使用 AskUserQuestion 询问："提取结果是否准确？如有遗漏请补充。"
   - 选项 A："准确，继续回写" → 进入步骤 5
   - 选项 B："有遗漏/错误，我来补充" → 用户补充后修正提取结果，再进入步骤 5

### 步骤 5：回写角色档案

对每个有变化的角色：

1. 读取 `{PROJECT}/bible/characters/{角色名}.md`
2. **已知信息更新**：追加到「后续获取」部分，格式：`第{N}章：{信息内容}（获取方式：{方式}）`
3. **状态变化**：更新存活状态/情绪/立场字段，追加变更记录
4. **关系变化**：更新关系网表格中对应行的"演变方向"
5. **弧光推进**：如有重大转折，更新动机与弧光中的"转折事件"
6. **新角色**：用 `assets/bible-templates/character.md` 模板创建新文件，填充提取的信息
7. 更新 `characters/_index.md` 总表

### 步骤 6：回写伏笔

1. 读取 `{PROJECT}/bible/foreshadowing.md`
2. **新伏笔**：追加新条目，编号递增（如上一个编号 F005，新条目 F006），状态=待回收
3. **回收伏笔**：更新对应条目的状态字段（待回收→已回收/已推翻/部分回收）
4. 写回 `foreshadowing.md`

### 步骤 7：回写数值

1. 读取 `{PROJECT}/bible/progression.md`
2. 更新当前积分/技能等级/道具库存的数值
3. 在变动记录表追加所有变动行
4. 写回 `progression.md`

### 步骤 8：回写时间线

1. 读取 `{PROJECT}/bible/worldbuilding/timeline.md`
2. 追加本章的时间线事件条目
3. 写回 `timeline.md`

### 步骤 9：回写副本事实（规则怪谈专属）

1. 读取 `{PROJECT}/bible/instances/{当前副本}.md`
2. 在「已建立事实」表追加新事实行
3. 写回副本文件

### 步骤 10：回写 Lorebook

1. 读取 `{PROJECT}/bible/lorebook.md`
2. 追加新关键词映射行
3. 写回 `lorebook.md`

### 步骤 11：更新写作进度

1. 读取 `{PROJECT}/tracking/progress.md`
2. 更新已完成章节列表和当前状态
3. 写回 `progress.md`

### 步骤 12：进入 S5

回写全部完成后，自动通知用户："事实提取+回写完成，接下来自动执行 S5 一致性审查"
自动进入 S5 流程。

## S5: 一致性审查

**触发条件**：S4 事实提取+回写完成后自动接续，或用户说"小说审查"/"一致性检查"时独立执行。

### 步骤 1：加载类型配置

1. 读取 `{PROJECT}/bible/genre-config.md`
2. 提取 `check_priority` 列表（含每层的 name、level、sources、description）
3. 如 genre-config.md 不存在，使用默认通用校验（仅人物性格+伏笔状态两层，均为 SOFT）

### 步骤 2：读取新章节

1. 读取 `{PROJECT}/chapters/ch{N}.md`（刚定稿的正式版）
2. 读取 `{PROJECT}/tracking/consistency-log.md`（历史审查记录，避免重复报已处理问题）

### 步骤 3：按优先级逐层校验

读取 `references/consistency-check-rules.md`（本 Skill 的审查规则），然后对 check_priority 中每一层按顺序执行：

#### 3.1 规则不违背（规则怪谈 HARD）
1. 读取 `{PROJECT}/bible/core-rules.md` 和 `{PROJECT}/bible/instances/{当前副本}.md`
2. 提取所有规则条文
3. 扫描新章节中每个角色行为/事件
4. 判断是否有行为/事件直接违反某条规则
5. 特殊判定：
   - 角色找到规则漏洞/第三种解法 → 不违规，但记录创意解法
   - 规则本身是误导规则，角色违反误导规则而遵守真实规则 → 不违规
   - 规则冲突优先级 → 按 core-rules.md 中定义的优先级判定

#### 3.2 信息一致性（HARD）
1. 读取出场角色的 `{PROJECT}/bible/characters/{角色名}.md`
2. 提取每个角色的「已知信息集」（开局已知 + 后续获取）
3. 扫描新章节中每个角色的推理/判断/决策
4. 交叉比对：角色是否基于已知信息集中不存在的信息做了判断？
5. 特殊判定：
   - 凭直觉/猜测碰巧正确（逻辑链不依赖未知信息） → 不违规
   - 本章刚被告知新信息并使用 → 不违规（S4 应已提取）

#### 3.3 数值连续性（SOFT）
1. 读取 `{PROJECT}/bible/progression.md`
2. 比对章节中涉及的数值描写与前章末状态
3. 积分/道具数量开头与上章末不一致 → 记录差异
4. 技能等级凭空提升无对应事件 → 记录差异

#### 3.4 轮回记忆一致性（规则怪谈 SOFT）
1. 读取 `{PROJECT}/bible/worldbuilding/timeline.md` 轮回结构
2. 扫描角色记忆引用是否符合轮回设定（保留/清除）
3. 设定"保留记忆"但角色遗忘 → 记录差异
4. 设定"清除记忆"但角色引用上一轮信息 → 记录差异

#### 3.5 伏笔状态（SOFT）
1. 读取 `{PROJECT}/bible/foreshadowing.md`
2. 检查预计回收章≤当前章的"待回收"伏笔是否已回收
3. 未回收 → 提醒（SOFT放行）
4. 新伏笔与已有伏笔功能重复 → 提醒

#### 3.6 人物性格漂移（所有类型 SOFT）
1. 读取出场角色档案的性格三词和锚定说明
2. 扫描角色言行是否偏离三词锚定
3. 完全矛盾且无合理解释 → 记录差异
4. 有弧光推进解释 → 不违规

> 注：以上 3.1-3.6 为规则怪谈的 6 层校验。其他类型按各自 genre-config.md 的 check_priority 配置执行对应层。如末世生存的重点是物资/基地数值连续性，都市异能的重点是异能使用边界，悬疑推理的重点是线索逻辑链闭合。

### 步骤 4：记录审查结果

将所有校验结果写入 `{PROJECT}/tracking/consistency-log.md`，追加本章的审查记录：

```markdown
## 第 {N} 章一致性审查

审查时间：{当前时间}

| 优先级 | 校验层 | 结果 | 矛盾描述 | 建议修复 |
|--------|-------|------|---------|---------|
| HARD | {层名} | ✅通过 / ❌违规 | {描述} | {建议} |
| SOFT | {层名} | ✅通过 / ⚠️差异 | {描述} | {建议} |
```

### 步骤 5：硬违规拦截

如任何 HARD 层校验结果为 ❌违规：

1. **中止流程，不进入下一章**
2. 标记当前章节需重写
3. 写入 `{PROJECT}/tracking/rewrite-history.md` 记录违规详情
4. 向用户输出警告：

```
⚠️ 第{N}章存在硬违规，必须修复才能继续！

违规详情：
{逐条列出HARD层违规的具体描述和建议修复方案}

建议操作：
A. 重写本章（自动触发 S7 修订流程）
B. 调整设定（修改 core-rules.md 或角色已知信息等 bible 文件）
C. 手动修改章节文本后重新跑 S4+S5
```

5. 使用 AskUserQuestion 让用户选择修复方式
6. 如选 A → 触发 S7 修订重写（如 S7 已实现），或手动回到 S3 重写
7. 如选 B → 用户修改设定后，重新从 S2 开始注入并重写
8. 如选 C → 等用户修改完后重新执行 S4+S5

### 步骤 6：软违规放行

如所有 HARD 层通过，仅 SOFT 层有差异/提醒：

1. **继续流程，允许进入下一章**
2. 向用户输出审查摘要：

```
📋 第{N}章一致性审查结果

硬校验（一票否决）：
  ✅ {各HARD层结果摘要}

软校验（记录放行）：
  ✅/⚠️ {各SOFT层结果摘要}
  （差异已记录到 consistency-log.md，可后续统一修订）

总判定：✅ 通过，可继续下一章
```

3. 更新 `{PROJECT}/tracking/progress.md` 的当前状态为"第{N}章完成，可继续"
4. **大纲续展检查（v2 新增）**：读 `outline.md`，统计"待写作"章节卡数量
   - 若 ≤ 2 → 先执行 `references/outline-extension.md` 续展下一段落（用户确认后），再继续
   - 若全书已完成（已完成=总预计）→ 提示用户可进入 S8 全局终审，不再续展
5. 更新 `run-state.md`：当前章节=第{N}章、阶段=空闲待命、下一动作=写第{N+1}章（或续展/终审）、累计统计+1
6. 询问用户："是否继续写第{N+1}章？" → 如确认，自动触发下一轮 S2→S3→S4→S5 循环

## S6: 场景扩写

**触发条件**：用户说"扩写"/"展开这段"/"详细描述"/"展开第X章某段"时触发，取 Sudowrite Expand 之长。

### 步骤 1：定位扩写目标

1. 从用户指令中确定扩写目标：
   - 指定了章节号+段落 → 直接定位 `{PROJECT}/chapters/ch{N}.md` 中对应段落
   - 只说"扩写"无具体指向 → 使用 AskUserQuestion 追问："请指定要扩写的章节和段落范围（如'第3章第2段'或'第5章战斗场景'）"
   - 指定了章节号但无段落 → 读取该章节全文，展示段落索引，让用户选择
2. 读取 `{PROJECT}/chapters/ch{N}.md` 定位目标段落原文
3. 确定扩写倍率：
   - 用户指定了目标字数 → 按目标字数扩写
   - 未指定 → 默认将目标段落扩写至原字数的 3-5 倍
   - 使用 AskUserQuestion 确认："将这段约{原字数}字的内容扩写至约{目标字数}字，是否合适？"

### 步骤 2：加载相关设定

1. 读取 `{PROJECT}/bible/meta.md` — 确定故事基调
2. 读取目标段落中涉及角色的档案 `{PROJECT}/bible/characters/{角色名}.md` — 性格/口癖/已知信息/关系
3. 如涉及特定地点，读取 `{PROJECT}/bible/worldbuilding/locations.md` 中对应段落 — 环境描写参考
4. 如涉及副本/规则，读取 `{PROJECT}/bible/instances/{副本}.md` — 副本场景和规则约束
5. 读取 `{PROJECT}/bible/style-guide.md` — 文风/描写侧重/禁用词
6. 读取 `{PROJECT}/bible/lorebook.md` — 匹配段落中关键词，注入关联设定

### 步骤 3：组装扩写提示词

1. 构建"扩写指令提示词"结构：

```
【扩写指令】
原文段落：
{目标段落原文}

扩写目标：将上述内容展开为约{目标字数}字的完整场景

【故事基调】{meta.md 一句话核心}
【出场人物】{每人：性格三词+口癖+标志动作+当前情绪+已知信息}
【场景设定】{地点环境描写参考}
【当前副本规则】{如适用，副本规则集}
【文风要求】{风格指南要点 + 描写侧重}
【扩写维度要求】
- 环境描写：{根据 style-guide.md 的描写侧重调整比重}
- 对话：{根据对话风格要求}
- 动作：{节奏偏好}
- 内心：{叙事距离决定比重}
- 感官：{五感描写比重}
【禁用表达】{禁用清单}
【约束】严格遵守所有规则和已知信息，不得引入与原文矛盾的新事实
```

2. 根据类型定制扩写侧重：
   - 规则怪谈 → 环境氛围（诡异感）> 规则暗示 > 心理恐惧 > 动作
   - 末世生存 → 动作（生存操作细节）> 环境荒废感 > 物资描写 > 心理
   - 都市异能 → 动作（异能战斗）> 心理（能力体验）> 环境 > 对话
   - 悬疑推理 → 心理（推理过程）> 环境线索 > 对话交锋 > 动作
   - 通用 → 按用户 style-guide.md 的描写侧重

### 步骤 4：扩写生成

1. 使用组装的扩写提示词生成长场景
2. 扩写内容必须包含：
   - 环境描写（场景氛围、光影、声音、气味）
   - 角色对话（符合各角色口癖和性格三词）
   - 角色动作（符合能力限制和当前状态）
   - 角色内心（符合叙事距离设定）
   - 感官细节（视/听/触/嗅/味，按描写侧重分配）
3. 扩写不得改变原文的核心事件走向，只能增加细节层次
4. 扩写不得引入原文没有的新角色/新规则/新重大事件

### 步骤 5：用户审阅

1. 将扩写结果与原文对照展示（原文 → 扩写版）
2. 使用 AskUserQuestion 询问："扩写完成，是否满意？"
   - 选项 A："满意，替换原段落" → 进入步骤 6
   - 选项 B："需要调整细节" → 用户指出调整点，微调后重新展示
   - 选项 C："不满意，重新扩写" → 回到步骤 4（最多重试 2 次）

### 步骤 6：替换原段落并更新章节

1. 读取 `{PROJECT}/chapters/ch{N}.md` 全文
2. 用扩写版替换目标段落
3. 写回 `ch{N}.md`（更新定稿时间标注）
4. 向用户展示更新后的章节全文（或仅展示修改段落 + 前后文衔接）

### 步骤 7：事实提取回写（如有新事实）

1. 比对扩写版与原文，识别扩写中新增的细节
2. 判断是否产生了需要回写到 bible 的新事实：
   - 新的感官细节/环境描写 → 通常不需要回写（纯描写层）
   - 新的角色行为/决策 → 需要回写（可能影响已知信息/状态）
   - 新的伏笔暗示 → 需要回写（应登记到 foreshadowing.md）
3. 如有需要回写的新事实，执行 S4 步骤 3-11 的提取和回写流程（只提取新增事实，不重新提取全章）
4. 如无新事实，跳过此步骤

### 步骤 8：一致性审查（可选）

1. 使用 AskUserQuestion 询问："扩写后是否需要跑一次一致性审查？"
   - 选项 A："需要" → 对第{N}章执行 S5（只审查扩写段落相关层）
   - 选项 B："不需要，继续写作" → 结束

## S7: 修订重写

**触发条件**：S5 硬违规拦截时自动触发（用户选"重写本章"），或用户主动说"重写"/"修改第X章"/"修订"/"rewrite chapter"时触发，取 Sudowrite Rewrite 之长。

### 步骤 1：读取违规/修改记录

1. **S5 自动触发时**：
   - 读取 `{PROJECT}/tracking/consistency-log.md` 中本章的硬违规条目
   - 读取 `{PROJECT}/tracking/rewrite-history.md`（如有历史重写记录）
   - 提取违规详情：校验层名/矛盾描述/建议修复方案

2. **用户主动触发时**：
   - 使用 AskUserQuestion 追问："请说明需要修订的内容（如'第3章主角违反了规则X'或'第5章对话需要修改'）"
   - 或用户已在指令中说明 → 直接记录修改需求
   - 如用户只说"重写第X章"无具体说明 → 读取该章节全文，展示并追问："需要修改哪些段落/方面？"

### 步骤 2：定位修改范围

1. 读取 `{PROJECT}/chapters/ch{N}.md` 全文
2. 根据违规/修改需求确定修改范围：
   - **局部重写**：只修改违规段落/用户指定段落（保留其他段落不变）
   - **全章重写**：如违规涉及多处或贯穿全章，重写整章
3. 使用 AskUserQuestion 确认修改范围："根据违规/修改需求，建议{局部/全章}重写。是否同意？"
   - 选项 A："局部重写，只改问题段落" → 局部模式
   - 选项 B："全章重写" → 全章模式
   - 选项 C："我来指定具体段落" → 用户指定范围

### 步骤 3：重新注入上下文（S2 流程）

1. **完整执行 S2 的全部 10 个步骤**，重新组装上下文提示词
2. 在组装的上下文提示词中追加"修复指令段"：

```
【修复指令】★★★必须优先执行
本章存在以下问题需要修复：

1. {违规层名}：{矛盾描述}
   修复要求：{建议修复方案}

2. {违规层名}：{矛盾描述}
   修复要求：{建议修复方案}

请在撰写时优先解决以上问题，确保修复后不再出现相同矛盾。
```

3. 将修复指令段插入到上下文模板的"本章定位"段之后，"出场人物"段之前

### 步骤 4：执行重写

**局部重写模式**：

1. 保留无需修改的段落原文
2. 只对需要修改的段落重新撰写，带入 S2 上下文 + 修复指令
3. 确保重写段落与前后保留段落的衔接自然（语气/节奏/信息连续）
4. 不得在重写段落中引入保留段落中没有的新重大事件/新角色

**全章重写模式**：

1. 使用完整的 S2 上下文 + 修复指令重写整章
2. 尽量保留原文中无问题的精彩段落/表达（不是完全重新生成，而是"修复版"）
3. 确保修复了所有硬违规问题

### 步骤 5：展示重写结果

1. **局部重写**：展示修改段落的新旧对比（旧版 → 新版），以及前后衔接处
2. **全章重写**：展示完整新章节，标注与旧版的主要差异点
3. 使用 AskUserQuestion 询问："重写完成，是否确认？"
   - 选项 A："确认，替换原章节" → 进入步骤 6
   - 选项 B："还需调整" → 用户指出调整点，微调后重新展示
   - 选项 C："不满意，重新重写" → 回到步骤 4（最多重试 2 次，超过后建议用户手动调整设定）

### 步骤 6：更新章节文件

1. 将重写确认后的内容写入 `{PROJECT}/chapters/ch{N}.md`
2. 更新文件头标注：

```
> ✅ 正式版本 — 第{N}章定稿（修订版）
> 定稿时间：{当前时间}
> 修订原因：{S5硬违规 / 用户要求修改}
> 状态：final
```

3. 将旧版章节备份到 `{PROJECT}/tracking/rewrite-history.md`（记录旧版关键段落 + 修订原因 + 修订时间）

### 步骤 7：重新执行 S4 事实提取+回写

1. 对修订后的章节重新执行完整的 S4 流程（步骤 1-12）
2. 注意：
   - 已在旧版中提取过的事实 → 如修订版中仍然存在，保留回写结果不变
   - 旧版中有但修订版中删除的事实 → 从 bible 中移除对应回写（如删除了某个伏笔埋设，则从 foreshadowing.md 中删除该条目）
   - 修订版中新增的事实 → 按正常流程回写
3. 这个步骤确保 bible 与修订版章节保持同步

### 步骤 8：重新执行 S5 一致性审查

1. 对修订后的章节重新执行完整的 S5 流程（步骤 1-6）
2. 如再次出现硬违规：
   - 记录到 rewrite-history.md（标注为"二次违规"）
   - 使用 AskUserQuestion 询问："修订后仍存在硬违规：{描述}。是否继续修订？"
     - 选项 A："继续修订（触发新一轮 S7）" → 回到步骤 1（最多累计 3 轮重写）
     - 选项 B："调整设定（修改 bible 文件）" → 用户修改设定后，从 S2 重新开始
     - 选项 C："手动处理" → 结束流程，等用户自行解决
3. 如 S5 通过 → 流程完成，可继续下一章

### 重写次数限制

- 同一章节最多累计 3 轮 S7 重写
- 超过 3 轮后强制停止，提示用户："第{N}章经3轮重写仍未通过一致性审查。建议检查 bible 设定是否有内在矛盾，或手动修改章节文本。"
- 每轮重写都记录在 `rewrite-history.md` 中，包含：轮次/违规描述/修复方案/结果

## S8: 全局终审（v2 新增，全书完成后触发）

**触发条件**：用户说"全局审查"/"全书终审"/"final audit"/"一致性总检"；或 `progress.md` 显示"已完成章节 = 总预计章节"且用户确认收尾；或用户指定"审查第X-Y章"。

> 与 S5（逐章）互补：S8 是**跨章节、跨 bible 的全局扫描**，专门捕获只有"看完全书"才能发现的系统性问题——孤儿伏笔、跨章信息泄漏、规则漂移、数值时间线断点、弧光断裂。

### 步骤 1：加载全集

1. 读取 `bible/` 全部模块（meta / genre-config / outline / characters/* / worldbuilding/* / instances/* / foreshadowing / progression / lorebook / style-guide）
2. 读取 `chapters/` 全部正式版章节（按文件名排序）
3. 读取 `tracking/consistency-log.md`、`tracking/rewrite-history.md`、`tracking/global-audit-log.md`（历史）

### 步骤 2：按 10 个全局维度扫描

读取 `references/global-audit-rules.md`，逐维度执行：

1. **孤儿伏笔**：`foreshadowing.md` 中状态=待回收/部分回收且预计回收章 ≤ 全书末章却不见回收 → 必须处理
2. **数值时间线完整性**：以 `progression.md` 变动记录重建全书曲线，查断点（某章末值≠下章开头值且无事件）
3. **信息时间线完整性（角色知情权）**：按各角色「已知信息集→后续获取」时间戳重建"第K章他知道什么"，回扫全书查跨章信息泄漏（S5 单章视角的盲区）
4. **角色弧光连续性**：弧光走向是否在全书事件中得到支撑，有无中途人设突变
5. **规则跨章自洽**：后文是否悄悄改了前文确立的规则（规则漂移）；冲突优先级是否一贯
6. **时间线/行程合理性**：多线叙事时间戳冲突、角色能否同时出现在两地
7. **伏笔密度与节奏**：统计埋设/回收比，标记连续空白 or 单章过载（非错误，供优化）
8. **类型专属全局项**：规则怪谈轮回记忆一贯性 / 末世物资自洽 / 异能上限未突破 / 悬疑所有动机闭环
9. ★**角色出场连续性（v3 新增）**：扫描全书每章的角色出场列表，统计每个核心角色的"最长连续缺席章数"。阈值警戒：主角缺席 > 3 章、其他核心角色缺席 > 5 章 → 标记为"角色掉线"，建议补充桥段或交代去向
10. ★**命名一致性（v3 新增）**：从 bible（worldbuilding/locations.md、characters/*.md、core-rules.md）中提取所有专有名词（地名、人名、科技名、舰船名、组织名），建标准命名表。回扫全书章节正文，匹配是否存在同一实体的非标准变体（例如"沉默之巢"vs"沉默巢"）。标记差异，统一修正

### 步骤 3：生成全局审查报告

1. 写入 `tracking/global-audit-log.md`（格式见 `assets/tracking-templates/global-audit-log.md`）
2. 向用户输出摘要：

```
📑 全书一致性终审报告（共 {M} 章）

🔴 必须修复（阻断发布）：
  - [孤儿伏笔] F012 应在第40章回收，全书未见
  - [信息泄漏] 主角第12章使用了第25章才获得的信息
  - [规则漂移] 第30章违背了第5章确立的元规则R-03

🟡 建议优化（不影响发布）：
  - [节奏] 第18-22章伏笔推进空白
  - [弧光] 配角乙动机支撑不足

🟢 通过项：数值时间线连续 / 角色弧光闭环 / 规则跨章自洽

总判定：❌ 存在 {n} 项阻断问题 / ✅ 可发布
```

### 步骤 4：修复引导

- 对每项阻断问题给出具体修复建议 + 命中章节
- 修复路径：
  - A. 局部补写/修订 → 触发 S7（针对具体章节）+ S4+S5 重跑
  - B. 调整设定 → 修改 bible 后重跑受影响章节的 S2→S5
- 修复后建议再跑一次 S8 确认清零
- 更新 `run-state.md`：阶段=S8终审、待处理事项记录阻断问题
